## 1. 这个项目是什么

一个面向泰国编程初学者的错误学习系统。

**一句话：**
> ChatGPT 每天都在原谅同一个错误。我们记住它。

**核心闭环：**
```
学生贴报错 → AI 分类 + 泰语解释 + 因果回放
          → 写入 CSV
          → 数"这是第几次"
          → 针对性练习题
          → 老师看到全班聚合
```

## 2. 护城河（唯一不可复制的东西）

**"这是你第 3 次犯这个错了。"**

- 泰语解释、因果回放、练习题 → 都是提示词工程，一天可被复制
- **"第 3 次" → 靠时间积累的历史，抄不走**
- 老师视图、成本缓存 → 都是"记得"的下游产物

一次错误是噪音；三次同样的错误是**一个概念缺口在说话**。
产品的全部意义：**把噪音变成诊断。**

## 3. 明确不做的事（防止跑偏）

- ❌ 自动捕获 traceback（`sys.settrace` / `subprocess`）→ **手动粘贴就够**
- ❌ 通用执行可视化（不要重做 Python Tutor）→ 只做**出错时刻**的因果回放
- ❌ Postgres / 登录系统 / Docker / 部署 → **一个 CSV 足够**
- ❌ PPT 生成器 → 已砍
- ❌ 移动端 / 离线 → 一周做不完

## 4. 数据结构（先定死，收完数据再改字段 = 历史数据作废）

`errors.csv`

| 列 | 说明 |
|---|---|
| `timestamp` | 时间 |
| `student_id` | `student_01`（**绝不用真名**） |
| `error_type` | **必须来自下方封闭分类表** |
| `raw_error` | 原始 traceback |
| `code_snippet` | 学生代码（**别省这列**，老师视图靠它） |

## 5. 封闭分类表（产品的生死线）

AI 若自由描述错误类型，同一个错会被写成三种说法 → **频率统计永远失效 → 产品直接死掉。**

```python
ERROR_TYPES = [
    "变量未定义",
    "作用域问题",
    "返回值vs打印",
    "类型不匹配",
    "缩进错误",
    "索引越界",
    "循环逻辑错误",
    "函数调用参数错误",
    "字典/列表操作错误",
    "其他",
]
```

**规则：**
- 提示词里把这个列表写死，要求 AI 一字不差地返回其中一个
- 代码里加保险：返回值不在列表中 → 归为 `"其他"`
- 最终分类表用**收集到的 40 条真实报错**来定，不要凭空想

## 6. 提示词模板

```
你是泰国编程初学者的助教。

分析下面的代码和报错，只返回 JSON，不要任何其他文字、不要 markdown 代码块。

格式：
{
  "error_type": "只能从这个列表选一个，一字不差：[变量未定义, 作用域问题, 返回值vs打印, 类型不匹配, 缩进错误, 索引越界, 循环逻辑错误, 函数调用参数错误, 字典/列表操作错误, 其他]",

  "concept": "学生真正没学会的概念"
}

代码：
{code}

报错：
{error}
```

## 7. 分步实现（每步跑通才进下一步）

### Step 0 · API Hello World 【最优先，跑通前不写任何其他代码】
20 行：发一句话给 Claude API，打印返回。不做别的。

### Step 1 · 手动输入
```python
code = input("贴上你的代码：")
error = input("贴上你的报错：")
```
不要自动捕获。评委不在乎。

### Step 2 · 强制结构化输出 【最难，留足时间】
用上面的提示词，让 `json.loads()` 成功解析。

⚠️ AI 常在 JSON 外多写一句"好的，这是结果："→ `json.loads()` 炸掉。
必须写一个清洗函数：剥掉 ` ```json ` 围栏 + 只取第一个 `{` 到最后一个 `}`。
**这一步吃掉的时间最多，这是正常的。**

### Step 2.5 · Eval：验证分类器准不准 【最容易被跳过，但跳过后面全是垃圾】

**核心问题：你怎么知道 AI 分对了？**

如果分类准确率只有 50%，那么频率统计、"第 3 次"、老师视图、练习题——**全都是在统计噪音。**

**做法（半小时）：**
1. 收集到的 40 条真实报错，**自己用纸笔手工标注**正确类别 → 这就是 ground truth
2. 让 AI 跑一遍这 40 条
3. 数对了几条

```python
准确率 = 对的条数 / 40
```

就是一个除法。

**两个产出：**
- **工程上**：准确率低 → 知道要改提示词，而不是瞎改
- **Pitch 上**：「我们的分类器在 40 条人工标注的真实数据上达到 85% 准确率」
  —— 这是**可验证的**，比任何话术都有杀伤力

⚠️ **分类表必须从这 40 条真实数据里长出来，不是拍脑袋定的。**
手工标注时可能发现：「缩进错误」一条都没有，而「忘记冒号 `:`」占了三分之一。
**猜出来的分类表会毁掉频率统计。**

### Step 3 · 写入 CSV
追加模式写入五个字段。

### Step 4 · 数"第 3 次" 【护城河，就是一个 for 循环】
```python
count = 0
for row in rows:
    if row["student_id"] == sid and row["error_type"] == etype:
        count += 1
```
**到这一步，产品已经成立了。**

### Step 5 · 生成针对性练习
把该学生的 top 1 错误类型丢回 API：「针对这个错误出 3 道泰语练习题」。

### Step 6 · 老师视图
同一个 CSV，按 `error_type` 聚合 → 一句话 + 一张柱状图。
> "本周全班 68% 卡在「作用域」和「返回值vs打印」。"

### Step 7（有余力）· 缓存
相同 `error_type` + 相似代码 → 直接复用已有解释，不调 API。
Demo 时展示：**学生数在涨，API 调用次数走平。**
这是全部公益叙事的唯一证据。

## 8.5 学习模式（对 Claude Code 的固定要求）

**我是编程初学者，正在通过这个项目学习。请始终遵守：**

- **每一行加中文 + 英文注释**，解释**「为什么这样写」**，而不是「这行做了什么」
  - ❌ `# 打开文件` (open the file)
  - ✅ `# 用 'a' 而不是 'w'，否则每次写入会覆盖掉历史记录，护城河就没了`
    `# Use 'a' not 'w' — 'w' wipes history, which kills the whole moat`
- **用到我可能没见过的语法时，先停下来解释**，不要直接写进代码
- **不要一次给超过 30 行代码**——超过 30 行我只会滚动，不会阅读

**额外原则：**
- 我卡住时，**先给方向和提示，不要直接给答案**
- 我说出自己的理解时，**指出哪里错了**，不要直接纠正成正确版本
- 不能向别人解释的代码，**不许留在这个项目里**
